Payroll Intelligence Agent: defining what good means before the agent exists

In January 2026 the PM building the Payroll Intelligence Agent needed ground truth for the top 51 payroll questions — golden answers plus a tool and data inventory — and she needed it urgently. Her message, paraphrased: "You said you had this, but I'm not seeing anything." If we didn't deliver, design would stay an outside consultant to the agent build.

Role
Led the effort — mobilized 3 designers, then built the generator myself
Context
Ground truth for a payroll agent, before the agent existed
Scale
51 questions + 9 diagnostic troubleshooting topics
Outcome
Reusable blueprint, adopted and extended by the partner team

Why payroll punishes vagueness

Payroll is the worst possible place to wing it. It's regulated, so a wrong answer isn't just unhelpful — it can create a compliance problem. It's data-dependent, so the right answer changes with each account's setup. And it's emotionally loaded, because the question behind every question is "will my people get paid?" An agent operating here needs its definition of good written down before it answers anyone: what the correct answer is, what data it needs to give that answer, and when it must stop and hand off to a human.

That definition didn't exist for the top 51 payroll questions. The PM needed it to build. We had a week's worth of credibility to either earn or lose.

Run it like an operation, not a heroic all-nighter

I got leadership approval to commit to this full-time for the week, then mobilized three designers plus product and content partners. The leverage move: I didn't write the answers myself. I wrote the instructions and templates — what a complete golden answer looks like, which tool and data dependencies to capture, what format the agent team could actually consume — and held the line on scope, format, and timeline while the team produced. Leadership had suggested scaling the scope down. We delivered all 51 questions in about ten days, as a blueprint the agent team could keep extending rather than a one-off document.

The relationship changed shape with the deliverable. We went from consultants the agent team occasionally asked for opinions to embedded contributors with a seat in the build.

The harder half: questions without a single right answer

Then I took the follow-up nobody volunteered for: nine troubleshooting topics — auto-payroll blocked, payroll run twice, direct-deposit changes, and the rest. These don't have golden answers. They have golden decision trees, and writing them takes SME-level diagnostic coverage.

How-to vs diagnostic · reconstructed
How do I add an employee?
HOW-TO — one known path: the steps are the same for every customer. A golden answer is a verified procedure. ground truth = the correct steps
My employee is missing from this payroll run.
DIAGNOSTIC — no single answer exists. Is the employee active? Assigned to this pay schedule? Hired after the pay period started? The right response is a tree of checks, and the correct branch depends on what the system finds in that account. ground truth = a verified decision tree + the data each branch needs

Reconstructed example. How-to questions need golden answers; diagnostic questions need golden decision trees with every branch verified by a subject-matter expert.

Engineering the generator

Hand-writing nine SME-grade decision trees would have taken weeks I didn't have. So I engineered the Diagnostic Golden Dataset Generator: a ~27KB system prompt that generates diagnostic golden datasets in a strict, enforced structure. Every generated topic must produce eleven sections in a fixed order — omit none, combine none:

headerTitle, the exact customer question in quotes, named SMEs (or "[TBD — need SME validation]"), and reference URLs
questionThe customer question, phrased the way a customer types it
scopeWhat this topic covers and explicitly does not cover
sub-variantsDistinct flavors of the same complaint that need different handling
critical understandingWhat the agent must grasp about the domain before it acts
decision treeExplicit branching grammar: If [condition] → Go to Step X — no prose hand-waving
data · retrievableWhat the system can look up itself, per branch
data · must askWhat only the customer knows — each question must carry a Why: justifying that it can't be retrieved
reference contentThe grounding content each branch draws on
handoff triggersThe conditions under which the agent must escalate to a human
root-cause summaryThe map of causes the tree resolves, stated plainly

The structure alone wasn't enough — the prompt also had to control how the model reasons. It runs a six-step pre-generation reasoning process before writing anything, and it carries contrastive correct/incorrect examples so the model learns the difference between an answer and an answer-shaped sentence: "Check the settings" is wrong; "Go to Settings > Payroll > Tax Setup and verify the EIN field" is right. PII rules and a validation checklist gate the output. And nothing generated ships as-is: SMEs review and edit every draft — the generator does the structural labor, humans own the truth.

Where it went

The framework was adopted by the partner team and extended to new domains beyond the nine topics it was built for. Downstream, payroll agent quality lifted +9.8 points through the golden-dataset and evaluation program — that lift belongs to the quality program as a whole (case study 03), and this dataset is one of the things feeding it.

What this proves

Agent behavior design isn't styling an agent after engineering builds it. It's defining ground truth, tool and data dependencies, and escalation triggers before the agent exists — so the build has something true to aim at. And it shows the move I keep making: take an ambiguous, urgent ask, turn it into a system with instructions and templates, and hand back something other people can extend without me in the room.
