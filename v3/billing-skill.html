No. 01 · BillingSkill

The charge nobody could explain.

Subscription billing drives roughly 800K customer contacts a year. The assistant answered every one of them with the same help article.

RoleSolo build
WhenMarch – May 2026
StackAgent repo · Cursor · skill YAML · MCP
OutcomeWorking skill + 108-case eval, submitted and judged

The data to answer properly existed the whole time: the plan, the add-ons, the proration that turns $99 into $107. Nothing connected it to the conversation. I'd redesigned this support area once already, back when the best fix was a better article. This time the platform could act, and I wanted to be the one who made it.

It started seven weeks before the hackathon.

In late March I built a proof of concept on my own time: a prototype that looked up a subscription charge and explained it with the customer's actual account data. A screen recording, a short deck, a working idea.

I'd spent the year before defining what good agent behavior looked like for other teams: golden datasets, gating criteria, judge models. Underneath the prototype sat a bigger question. Could the person who writes the evaluation build the thing being evaluated?

So when the company announced an internal builder hackathon for May, I wasn't starting a weekend hack. I was starting the third act. The bet: a conversation designer who already shapes how the agent behaves can use the existing agent repo to build a working skill end to end.

Five days, day by day.

The hackathon window gave me five working days. Here's what each one held.

DAY 1Scope from evidence

Analyzed 500+ labeled customer utterances and defined 7 intent categories. Chose subscription charges as the target: highest volume, clearest data dependency.

DAY 2Write the skill, build the tool

Wrote the skill YAML: tone, when-to-use conditions, graceful handling, and an admin vs non-admin permission check. Built a billing lookup tool with three modes, LIVE for real APIs, EVAL for scenario injection, DEV for mock data, and registered it in the agent repo.

DAY 3Test end to end

Ran the skill locally against the live agent through a Streamlit harness. Validated both permission paths: admins get the charge detail, non-admins get a safe redirect.

DAY 4Hit the wall, write the wall down

Live billing-history APIs answered with a permissions block. Documented the blocker precisely for the platform team, then built the first 33-row golden dataset so evaluation wouldn't wait on access.

DAY 5Real conversations widen the map

Pulled 55 production conversations and rewrote the skill instructions for what people actually say: cancel intent, price-increase complaints, downgrades. The eval set grew to 108 cases across 13 intent categories.

The skill shipped with its own eval.

Every row in the golden dataset carries three things: a scenario, the mock account data the tool returns, and gating criteria written so a judge model can check them without a human in the room. This is one row. Run the judge yourself.

The set also caught the question hiding inside the question. When a billing complaint is really a cancellation, the skill offers a confirmation widget and a chance to fix the actual problem first. And before anything could collide with production behavior, I ran an overlap analysis against the 7 existing skills around billing.

How I built it.

I forked the agent repo and worked in Cursor with Claude as my pair: reading how existing skills were structured, writing the code I needed, testing everything on my machine. I learned MCP from scratch along the way; that piece was the two of us vibe-coding a local prototype. I documented the API blockers precisely and handed them to the platform team.

Building this way is a craft of its own: knowing what to ask the pair, what to verify yourself, and when to stop generating and read the code. It's what turned a year of defining quality into working software in five days.

What came of it.

I submitted on day five and the entry was judged alongside everything else. The skill, the three-mode tool, and the 108-case eval put the argument I'd been making all year into working form: the people who define what good looks like can build at the source.

"I went, I completed the task, and I'm proud our team was represented." from my note to the team · May 2026
