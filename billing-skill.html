The receipt

A large share of billing complaints are cancellations and downgrades wearing a billing question. The skill catches them, reads the account, gives the real steps, and offers a person. Execution stays out by design: the tools are read-only, and payment capture never enters the channel. An answer that is wrong costs a retry; a cancel that is wrong costs the account. Scoped against the billing skills already shipping, this one detects and routes. It never owns the cancel.

02The build

Spec, tool, server.

Scope came from evidence: 500+ labeled customer utterances, grouped and ranked, and one target chosen for volume and data dependency: subscription charges. The question the assistant kept deflecting was the one the account could already answer.

The skill and the tool

A skill spec first: tone, when-to-use conditions, graceful handling, and an admin check, so admins get charge detail and everyone else gets a safe redirect. Behind it, a billing lookup tool with three modes: live for the real API, eval for scenario injection, dev for mock data. Tested end to end through a local harness, both permission paths.

The move to MCP

The first version ran as tools inside the agent repo. That proved the behavior, not the path: in production the platform reaches its tools over MCP. So I learned the protocol from scratch, stood up a local MCP server, registered it, and watched the agent discover the tools and call them end to end.

03The constraint

Mid-build, the live API said no.

The live billing API answered with a permissions block, and access sat weeks deep in someone else’s queue. The build could stall on that, or stop depending on it.

The call · decouple

Evaluation stopped depending on access. The eval ran on scenario injection and mock data, the live mode stayed stubbed behind a precisely documented blocker, and the first golden set, 33 rows, went in the same day. Nothing waited.

What I did not build

The live data connection. The skill, the server, and the eval are proven end to end on mock data; wiring the real systems is engineering work, and it is scoped, named, and waiting on access. That is the honest line.

04The repair

Where I caught the machine.

Building with an AI pair is a discipline of its own: knowing what to verify yourself, and when to stop generating and read the code. The bug that proved it was invisible from inside the pair. The skill loaded clean. The server ran. And not one tool call arrived.

The same testing pass caught two more. The instruction to call the tool first was being read as call it after the customer confirms. And in the cancel flow, “no thanks” sent customers back into the cancel path, because the decline still contained the word cancel. Found by testing, named, fixed. The thinking was mine; the typing was shared.

05The eval

It came with its own eval.

Every row in the golden dataset carries three things: a scenario, the mock account data the tool returns, and gating criteria written as a rubric an LLM judge scores. The set started at 33 rows, scoped to subscription charges. Then production conversations reset the map: cancel intent, price-increase complaints, downgrades. The set grew to 108 cases across 13 intent categories, and the skill grew from an answer engine into an intent router.

Trusting the judge is its own discipline. Validating judges against human labels is Case 03, the Quality Program.

06The handoff

What it proves, and what is left.

The skill, the tools, the MCP server, and the 108-case eval run end to end on mock data today. What is left is production: wire the live data connection, harden the admin check from a prompt rule into the tool itself, and read the results on real traffic. I scoped that work; it needs engineering.

Shipped, I would measure it on the outcome, not the answer: first-turn resolution on billing questions instead of a redirect to an article, the share of cancellations caught while they still look like billing complaints, and a confidently-wrong rate held at zero. It goes out behind the eval as a gate, small and watched, before it runs wide.

The argument the build settles: the people who define what good looks like can build it. I designed the behavior, wrote the eval, made the boundary call, stood up the server, and fixed it when it broke.
