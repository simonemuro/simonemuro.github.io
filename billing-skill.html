01Background

The pivot, the mandate, and my element.

The charge-lookup case ends where this one begins: the MVP moved the experience, but the target answer needed billing APIs. By then Intuit was moving support into a unified agentic platform.

The company also asked designers, PMs, and managers to become builders. I used that mandate to learn the successor platform and turn the billing answer into a working skill.

02The problem

A real answer needs data the agent could not touch.

The target answer has to name the charge, what changed, and the math behind it. That means plan, add-ons, promotional dates, and recent charges for that specific account.

The agent had no billing tool to call. Without data, every reply was either generic, the article problem all over again, or guessed. A guessed number about someone’s money is worse than no answer.

So the problem was not writing the reply. It was giving the agent something true to say it with, inside boundaries that hold.

03Why it mattered

From a roadmap slide to a running agent.

The PRD proved the target answer was worth pursuing. It could not prove the answer was buildable: safe boundaries, real account data, verifiable behavior.

A working skill is a different argument. It turns get buy-in into watch it run, and it forces every behavior decision to survive contact with the code.

04Exploring solutions

Skill plus server, and the wall we hit.

On the agentic support platform, an experience is a skill: a spec that tells the agent how to behave, backed by tools it can call. So the shape was clear early. A billing skill, and behind it one lookup tool the agent calls for account data. The boundary call came first: the tool is read-only. It can read plans, add-ons, and charges. It cannot change a subscription, and payment capture stays out of the channel entirely.

Then the wall. Mid-build, the live billing API answered with a permission block, and access sat weeks deep in someone else’s queue. I did not wait. I designed the lookup tool with three modes: live for the real API, eval for scenario injection, dev for mock data. Evaluation stopped depending on access. The live mode stayed stubbed behind a documented blocker, and nothing else stopped.

05Design decisions

The conversation design itself.

Four decisions define the answer. Each one is visible in the recreated conversation in part seven.

Customer scanAnswer first.

The first sentence names what the charge is and why it changed before showing details.

TrustShow work quietly.

The tool trace is available, but collapsed so the customer is not forced through internal reasoning.

BoundaryRead, do not change.

The skill explains and routes. Subscription changes and payment capture stay outside the agent.

  1. 5.1

    Answer first, then the detail.

    The first sentence names the charge and why it changed. The breakdown rows come after, for anyone who wants to check the math. Nobody has to scroll to learn the answer.

  2. 5.2

    The reasoning collapses to one line.

    The lookups the agent ran sit behind a quiet Work done line. A customer can open it and see what was checked. Nobody is made to read it.

  3. 5.3

    Hand off when the exit shows.

    When a billing question is really a cancellation decision, the skill reads the account, gives the real steps, and offers a person. A specialist can save an account. A chatbot arguing cannot.

  4. 5.4

    There are things the agent must never do.

    It never cancels for you. The handoff card closes with two choices, talk to a specialist or keep my plan, and neither one is cancel. And every amount the agent states must trace to account data. No guessed numbers.

06The build

Skill, server, eval gate.

The skill is a spec before it is code: tone, when to use, graceful handling, and an admin check, so admins get charge detail and everyone else gets a safe redirect. Behind the spec sits the billing lookup tool. In production the platform reaches its tools over MCP, so I learned the protocol from scratch, stood up a local MCP server, registered it, and watched the agent discover the tools and call them end to end.

It broke once in a way worth telling. The skill loaded clean. The server ran. And not one tool call arrived.

The eval came with the skill, not after it. It started at 33 subscription-charge rows, then 55 production conversations reset the map: cancel intent, price complaints, downgrades. The set grew to 108 cases across 13 intent categories.

The eval is the gate. A change that fails a row does not ship.

08Learnings

What I learned, and what I would do differently.

Production conversations beat my assumptions. My first 33 eval rows tested the skill I had planned. The 55 real conversations showed the skill customers needed, and it was a different one. Next time I start with the transcripts.

An AI pair speeds up the typing, not the judgment. The config bug was invisible from inside the pair; I found it by stopping and reading the code. Knowing what to verify yourself is the actual skill.

Never let evaluation depend on access. The three-mode tool design came out of a blocked API, and it is the part of the build I would keep exactly as it is.

Two things I would do differently. I would request API access on day one, not when the build hit the wall. And I would push the admin check down into the tool itself, so the boundary lives in code instead of instructions.
