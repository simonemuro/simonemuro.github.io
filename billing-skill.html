No. 01 · BillingSkill

I built the thing I'd been grading.

For a year I defined what a good agent answer looks like, then handed the spec to the teams who build. Billing was where I built it myself. It's a top contact driver, and the assistant kept answering with the same help article, even though the data to give a real answer was right there.

RoleSolo build, AI-paired
WhenMarch – May 2026
StackAgent repo · Cursor · skill YAML · local MCP server
OutcomeWorking skill on a verified MCP server, 108-case eval, on mock data

The data to answer properly existed the whole time: the plan, the add-ons, the per-seat math that turns a $99 base into a full monthly bill. Nothing connected it to the conversation. I'd redesigned this support area once already, back when the best fix was a better article. This time the platform could act, and I wanted to be the one who made it.

Building was never the new part.

For years I built support experiences end to end, from idea to production: the chatbot and the voice assistants in the NLU era, then the hybrid systems when LLMs arrived.

In the agentic era the shape of the work changed. My team owned the behavior: the testing, the golden datasets, the end-to-end evaluation that decides whether an agent is good enough to ship. We defined the source of truth and handed it to the teams building the agents. Useful, but a spreadsheet of good, better, and best is not the thing itself.

There was a push across the company for everyone to build, and I was certain my design technologists could do more than define behavior. We could build it in the codebase: the agent instructions, the prompts, the tools, the skill, even a local MCP to stand in for the data. Billing was where I'd prove it. So I built the proof of concept myself, and used an internal hackathon to make it official.

The build, day by day.

The hackathon gave me five focused days to take it from prototype to a working, evaluated skill. Here's what each one held.

DAY 1Scope from evidence

I read 500+ labeled customer utterances, grouped them into 7 intent categories, and made the first call, subscription charges as the target, because it was the highest volume with the clearest data dependency.

DAY 2Write the skill, build the tool

Wrote the skill YAML: tone, when-to-use conditions, graceful handling, and an admin vs non-admin permission check. Built a billing lookup tool with three modes, LIVE for real APIs, EVAL for scenario injection, DEV for mock data, and registered it in the agent repo.

DAY 3Test end to end

Ran the skill locally against the running agent through a local test harness. Validated both permission paths: admins get the charge detail, non-admins get a safe redirect.

DAY 4Hit the wall, keep moving

The live billing APIs answered with a permissions block. So I made the call to decouple evaluation from access rather than let the build stall. I documented the blocker precisely and built the first 33-row golden dataset, so the skill could be proven on mock data while the real connection waited.

DAY 5Real conversations widen the map

I pulled real production conversations and let what people actually say reset the design, cancel intent, price-increase complaints, downgrades. That's the decision that grew the eval set to 108 cases across 13 intent categories.

What the research changed.

I'd scoped this as "explain the charge." Reading real conversations changed the shape of it. A large share of billing complaints were not really about the charge. They were cancellations and downgrades wearing a billing question. "Why is this so expensive" is often "I am about to leave."

So I redrew the skill from an answer engine into something closer to an intent router: read what the customer is actually trying to do, answer the billing question when that is what it is, and route the exit intents with care. That is where the eval grew from 33 rows to 108, and where the harder design lived.

Then I moved it to a real MCP server.

The first version ran as tools inside the agent repo. That proved the behavior, but it was not how the platform actually reaches its tools. So I made the call to rebuild it on MCP, the protocol the agents use to call tools, and I learned it from scratch.

I stood up a local MCP server, registered it with the agent, and watched the agent discover its tools and call them end to end. The data behind it is still mock, and the live connection is stubbed until engineering wires it to the real systems. But the path the skill takes in production is real, and I proved it works.

Where I caught the machine.

Building with an AI pair is a skill of its own: knowing what to ask, what to verify yourself, and when to stop generating and read the code. Two bugs stand out, both invisible from inside the pair.

The skill loaded, but no tool calls were reaching the MCP server. I traced it to two layers: the config named the server but left out the list of tools it exposed, so the agent registered none of them, and the instruction to call the tool first was being read as call it after the customer confirms. I fixed both and watched the calls land.

Then the cancel flow looped. The confirmation buttons were swallowed on rerun, and "no thanks" sent customers back into the cancel path because the decline still contained the word cancel. I found each by testing, named the cause, and fixed it. The thinking was mine; the typing was shared.

It came with its own eval.

Every row in the golden dataset carries three things: a scenario, the mock account data the tool returns, and gating criteria written as a rubric an LLM judge scores. Trusting that judge is its own discipline: validating it against human labels is the Quality Program. This is one row. Run the judge yourself.

The cancellation case is where the design earns its keep. When the intent is really an exit, the skill slows down: it reads the account, gives the real steps, and offers a person, but it never cancels for you. Before it could collide with what already ships, I made the scoping call myself. I ran an overlap analysis against the other billing skills and held this one to detect and route, not own the cancel flow.

How I'd know it worked, and what's left.

The skill, the tools, the MCP server, and the 108-case eval run end to end on mock data today. What's left is the production side: wiring the real data connection, hardening the admin gate from a prompt rule back into the tool, and a before-and-after read on real traffic. I scoped that work. It needs engineering, and that's the honest line.

If it shipped, I'd measure it on the outcome, not the answer: first-turn resolution on billing questions instead of a redirect to an article, the share of cancellations caught when they hide inside a billing complaint, and a confidently-wrong rate held at zero. I'd roll it out behind the eval as a gate, small and watched, before it ran wide.

What it already proves is the argument I'd been making all year. The people who define what good looks like can build it, not just spec it. I designed the behavior, wrote the evaluation, made the architecture call, stood up the server, and debugged it when it broke. The rest is engineering, and I know exactly where that line is, which is the next thing I want to be standing on the other side of.
