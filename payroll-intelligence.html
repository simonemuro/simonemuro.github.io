No. 02 · Payroll Intelligence

Teaching an agent what the experts know.

January 2026. A new payroll agent was being built, but the answers it needed didn't exist yet, at least not the verified ones. Ask a model a payroll question it can't look up and it will invent one, confidently. I built the framework that taught it what the experts actually know. The whole thing shows up in how it answers one question: where is my employee?

RoleAuthored the ground-truth framework: templates, instructions, the generator
ScopeThe high-volume questions + 9 troubleshooting domains
PaceFull set in about ten days
BuiltA system prompt that drafts diagnostic datasets for experts to correct

The agent already knows more than the expert.

"My employee is missing from this payroll run" sounds like one question. It's half a dozen root causes wearing the same sentence. A human expert opens with a long interview because a human starts blind.

The agent doesn't. Before its first reply it can already read the account: subscription status, employee records, pay schedules, the audit log. So the rule I wrote into the framework: don't mimic the expert's question sequence. Match the customer's state against the known root causes first, then ask one confirming question, and only for what only the customer can know.

Instructions, not answers.

Payroll is the worst place to wing it. It's regulated, so a wrong answer can create a compliance problem, not just a bad session. It's data-dependent, so the right answer changes with each account's setup. And behind every question sits the same one: will my people get paid?

The target was payroll's highest-traffic, highest-complexity questions, the ones an agent could resolve end to end, not just answer. One person hand-writing golden answers for all of them was fantasy. So I made the call that scales: I wrote the instructions instead of the answers, because instructions compound where answers don't. Templates for what a complete golden answer looks like, the tool and data dependencies to capture, the exact format the agent team could consume, and the gate each answer has to clear. One artifact, drafted so design, engineering, and quality could each pull the slice they needed from the same source of truth.

The full set landed in about ten days as a blueprint the agent team could keep extending, not a one-off document. I owned the framework and the prompts; the diagnostic knowledge inside them came from the experts who resolve these calls.

A generator for the hardest questions.

Golden answers work for how-to questions: one verified procedure, the same for every customer. Troubleshooting questions are different. They don't have golden answers; they have golden decision trees, and every branch needs expert-grade coverage.

Hand-writing nine of those would've taken weeks the agent team didn't have. So I built the Diagnostic Golden Dataset Generator: a system prompt that drafts the whole decision tree in one validated structure, so an expert corrects a draft instead of authoring from a blank page.

The prompt controls how the model reasons, not just what it outputs. It runs a six-step reasoning pass before writing anything, and it carries contrastive examples so the model learns the difference between an answer and an answer-shaped sentence: "check the settings" fails; "go to Settings → Payroll Settings → Tax Setup and verify the EIN field" passes. PII rules and a validation checklist gate the output.

Nothing generated ships as-is. Subject-matter experts review and edit every draft: the generator does the structural labor, humans own the truth.

And the generator is built to distrust the model. A control in the prompt bans payroll concepts the product doesn't actually have and locks the agent to a fixed set of verified tools, because the dangerous failure here isn't a wrong answer, it's a confident invented one. Every case also carries its own pass/fail gate: a handful of binary checks, each forcing the same question, can the agent infer this, or must it be told? I didn't just write the answer; I wrote the test that decides whether an answer is good enough to ship.

The knowledge lived in people, not the model.

The nine troubleshooting questions are umbrella subdomains, each one a whole territory of sub-questions, chosen by call volume, customer pain, and how long they keep experts on the phone. Auto-payroll blocked. Payroll run twice. Direct-deposit changes mid-cycle.

Filling the trees took about a week and a half of interviews with the people who actually resolve these calls: support, engineering, the policy team. A model asked for institutional knowledge will confidently invent it. The diagnostic paths had to come from humans.

Hand-built root causes age as the product changes, and a library of them is a real maintenance question at scale. The framework was the answer: structure the team regenerates as the product shifts, not a pile of documents to maintain by hand.

The deliverable encoded diagnostic knowledge that had lived only in experts' heads, in a form the agent team could keep regenerating. This is the shape of what I build now: the framework, the test that decides whether its output is good enough to ship, and the call on which questions to aim it at first.
