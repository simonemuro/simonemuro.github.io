No. 03 · The Quality Program

To show the way, you have to know the way.

For years I hand-built the help experience one turn at a time, in a system where a designer scripts every decision and the backend obeys fixed rules. Then the assistant became agentic: it plans, decides, and acts on its own. That changed everything about what "good" means, and about how you prove it before it ships.

My role: drove pre-launch quality for an agentic assistant inside an accounting product. I owned the eval rubric, the golden ground truth across words, actions, and charts, the simulator instructions, and the generators that scaled them.

I used to script every step. Now the system decides.

For years I crafted high-volume, high-complexity do-it-for-you help flows in a deterministic system: I scripted every turn, every decision, every action, and backend rules governed the rest. If a customer could reach it, I had drawn the path. Then we migrated those same flows to an agentic system that plans, decides, and acts on its own. The power is obvious. So is the risk: the path I used to draw, the system now draws for itself, live, on a real customer's books.

The hard part isn't teaching it to act. It's teaching it when not to.

An agent that can act is powerful and dangerous in the same breath. The job was to keep it reliable and inside its guardrails: to teach it to make good calls, take the right actions, and, just as often, to not act when it shouldn't. Some moves it can never make on its own. Cancelling a subscription or closing an account is always a question to answer, never an action to take. And anything irreversible, running payroll, deleting a record, updating a tax filing, has to stop and ask before it moves. A reversible action it can just do. An irreversible one it has to earn.

Before you can grade the agent, you have to know the way yourself.

You can't evaluate a path you haven't mapped. So with partners across support, customer success, and product, we wrote down, for each use case, the tools the agent may call, the data it may touch, the minimum it needs before it acts, the rules for reading a request, the boundaries it must never cross, and the gates that guard the high-stakes moves.

Then we defined what good, better, and best look like for each customer question. A ground truth isn't just a correct answer; it's the definition of what "good" means for one specific question. That written definition is the blueprint, and getting a cross-functional room to agree on it was most of the work.

Then you decouple the definition of good from the thing being graded.

The blueprint becomes a contract: a machine-readable golden dataset that holds the ideal path, the right words, the right actions, the right widgets, the right charts, independent of how the agent is actually built. That separation is the whole trick. It lets a judge take any live agent and ask three questions: is it following the path, where is it breaking, and why? The definition of good lives in one place; the implementation it grades lives in another.

Now you can measure the things a sentence hides.

A good answer used to be one judgment. For an agent it's a matrix. The judge scores the words, and then the things only a path reveals: did it take the right action, did it personalize to this customer, did it respect permissions and confirmations, and did it follow the right path with no wasted steps? Four new axes on top of the old ones. That's the difference between an answer that reads well and an agent that behaves.

The blueprint grades the system, and teams build to it.

Here's the part I didn't expect. A clear, shared, written definition of good turned out to be two tools in one. It's an evaluation instrument, the release gate every agent has to clear. But the same golden datasets became a build spec: teams use them to develop new features, building to the definition instead of just being graded by it.

The payroll-tax agent is where it gets concrete. Running payroll is irreversible, so the blueprint marks it high-stakes: the agent states exactly what it's about to do, shows a confirmation, and waits for a real "yes" before a single payment moves, and we test it against a copy of the books, never a customer's real ledger. The payroll skill's quality climbed sharply once we defined good that precisely, not because the model changed, but because we had finally written down what "right" meant.

A designer who builds the instrument, not just the design.

I owned this end to end: the rubric, the ground truth across words, actions, and charts, the simulated customer the judge tests against, and the generators that scaled it all, shipped as platform code through the same review pipeline as senior engineering. The discipline now spans the craft and the platform underneath it.

The honest catch, and the bug that mattered
The generators raised an honest problem: manual QA on a thousand pre-filled rows is the same work the pre-fill was meant to remove, so the QA had to be codified too, a dozen automated checks that fail the build like CI. One of them caught the bug that mattered: a path that could silently overwrite a human's original ground truth with model output, no audit trail, quietly poisoning every downstream judge. I rebuilt the write path so originals are byte-identical on every route, with a check that fails the run if it is ever violated. Fixed in about four hours.

What I'd do next

  • Push the blueprint earlier into design, not just before launch
  • Make the build-spec use as first-class as the eval use

The work was never grading answers. It was deciding what a good answer is for an agent that acts, then making that definition cheap to enforce on every release. To show the agent the way, we first had to know the way ourselves.
