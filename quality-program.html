Case No. 04 The Quality Program Intuit Intelligence

Every agent answer is graded against an answer key. Someone has to build the answer key.

Before a skill ships in Intuit Intelligence, it’s scored against golden datasets: rows that define what a good answer looks like, question by question. I built and led the practice that produces them, then turned the slowest part into a pipeline: generator prompts that draft like a senior labeler, a four-phase batch system with twelve codified QA checks, and a review loop where experts decide and machines never overwrite. It shipped through the engineers’ code review and merged.

01The problem

Three weeks per dataset, and the same errors every time.

A golden-dataset row isn’t a question and an answer. It’s the full definition of good behavior for one customer query, and every field of it has rules.

one golden-dataset row · anatomy field names condensed 
question: one real customer query
ground_truth_text: the answer, written the way the
  best labeler would write it
ground_truth_actions: the tool calls a done-for-you
  row must make, with inputs and outcomes
ground_truth_widgets: the confirmation widget an
  impactful action must show before it runs
ground_truth_charts: only when the data genuinely
  warrants a visual
user_simulator_instructions: how the simulated
  customer behaves, one rule per line, every line
  starting "If agent ..." or "After agent ..."
max_turn: instruction clauses + 1
early_stop_condition: the behavior that means the
  customer got what they came for

Condensed from the dataset standard I wrote; internal system names removed.

Authoring those rows by hand took about three weeks per dataset, and auditing more than a thousand existing rows showed the errors repeating in patterns: answers written into the wrong field, missing simulator instructions, mislabeled turns, formatting from the old schema leaking into the new one. The experts’ judgment was excellent. Their time was being spent on scaffolding.

02What gets graded

The judges score the conversation, not just the sentence.

At eval time, a user simulator plays the customer against the live agent, following the row’s instructions turn by turn, and LLM judges score the conversation against the ground truth. The rules the rows encode are specific about what counts.

03The generator

A prompt that drafts like the person it’s helping.

The fix for three-week authoring wasn’t a form-filler. I wrote a generator prompt that drafts each row the way the team’s best labeler would, then leaves the decision to a person. Its identity section is the design philosophy in one place.

generator prompt · identity section redacted excerpt 
You are a senior evaluation specialist and ground
truth architect for the Intuit Intelligence AI
evaluation pipeline. You are not a form-filler.
You are the most experienced human labeler on the
team.

You bring judgment to every row. A ground truth is
not just a correct answer. It is the definition of
what "good" looks like for a specific customer
question.

ACCURACY OVER COMPLETENESS. A missing chart is
recoverable. A wrong chart trains the evaluator to
score incorrect behavior as correct.

JUDGMENT OVER RULES. When you encounter a case that
does not fit a named pattern, ask: "What would the
best human labeler do here?" And do that.

SOURCE FIDELITY. The original ground truths were
validated by experts. Restructure and augment;
never rewrite or second-guess the content.

Redacted excerpt from the generator prompt I wrote; internal system and team names removed.

The generator reached version seven in about ten days, each version pushed by a real event: a run, a call, a piece of feedback from the experts who’d live with the output.

v1 · day one

First built: sixteen sections, using the labeler guide’s plain-text action format.

v2 · day two

Originals never modified. All machine output moves to staging columns; the expert’s original cells become untouchable.

v3

Skipped: an internal iteration that never left my desk.

v4

First live run on a 40-row sample, validating the confirmation-widget pattern and chart inference against real rows.

v5

A working call redefined what counts as a turn: one turn is one user message plus one response, and widget interactions don’t count. The generator’s turn math changed with it.

v6

Simplified after feedback that it did too much: tightened the done-for-you classification, removed rules that fought each other.

v7 · day ten

Column names aligned to the experts’ real working sheet, and a sign-off column nobody used was dropped. The version that ran on the full production sheet.

04The pipeline

Four phases, twelve checks, one green report.

Around the generator I built the batch pipeline: extract the sheet into 50-row batches, run one sub-agent per batch in parallel, assemble the results back into the sheet, then prove nothing broke. Experts spot-check samples before any full run. The proving is the part I codified: twelve QA checks that run after every assembly, and the run isn’t done until the report is clean.

validation run · 100-row sample

And the spot-check that matters most: the experts’ original ground truth text, byte-identical in 100 of 100 rows.

05The review

The review found the one path my guarantee could break on.

I’m a designer, and the pipeline shipped as a pull request into the engineers’ repository, with a real review. The comment that mattered pointed at a fallback for sheets without staging columns, where machine output could land in an expert’s original column.

code review · the comment

“Potential data loss on the fallback path”

"When the input sheet uses the plain schema, no staging columns, this function resolves the output to the original column... the original can be silently overwritten with the LLM's output. Consider either: an overwrite guard, an extended QA check, or documenting the fallback path as destructive-by-design."

The reviewing engineer was right, and a fallback path is a silent failure path in disguise.

the fix · same session

No fallback. No path to the originals.

Eliminate fallback path; auto-create recommended_* columns when missing, so originals are never written to on any path.

From the commit that closed the review: instead of guarding the dangerous path, the fix removed it. Sheets without staging columns now get them created automatically, with all twelve checks passing on both input shapes.

Condensed from the review record; reviewer’s name, ticket ids, and internal format names removed.

The staging-column pattern itself, machine drafts in one place and expert originals in another, came from a teammate on the quality team. What my pipeline added was making it a guarantee a machine checks on every run instead of a convention people remember. The review made even the guarantee stronger.

06The practice

A program is standards plus people who can apply them.

The pipeline was one piece of a practice. The rest was standards work: a five-step golden-dataset process for the org, a labeler guide covering the full row schema, and training designers and labelers on the grading rubric. Two pieces changed real decisions.

the standard

Simulator instructions got a grammar. Every line starts “If agent” or “After agent,” the turn limit derives from the clause count, and an early-stop condition says when the simulated customer is satisfied. Systemized across hundreds of rows, so any labeler’s rows drive the simulator the same way.

the audit

More than a thousand existing rows went through a keep-or-delete audit against the standard before anything new was generated.

the economics call

A 200-task human-labeled set validated the LLM judges, and once they held up, I cut a planned two-thousand-row vendor labeling dataset. Buying labels the judges could now produce would have spent budget proving what was already proven.

The practice outlived my part in it by design: the pipeline and its generator live in the team’s repository, shipped through their review, and the remaining generators and standards are documented for the people who run them now. The skills gated with these datasets include the ones in this portfolio: the payroll tax skills were graded against ground truth from this program, and the billing skill’s 108-case dataset was written to its standard.

Quality work scales when judgment is written down. That’s what the practice comes to: a prompt that reasons like the best labeler, a check that proves the expert’s work survived, a grammar any labeler can apply. The decisions that shaped all three were about what the machine must never be allowed to do.
