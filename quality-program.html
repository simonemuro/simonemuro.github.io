No. 03 · The Quality Program

The answer sounded right.

Agents on an agentic AI platform were taking real actions in customers' financial workflows. Sounding right isn't a quality bar for someone's books.

RoleLed the quality methodology and tooling
When2025 – 2026
InstrumentsGolden datasets · LLM judges · simulators · blind audits
OutcomeQuality became a release gate

A judge model can grade whether an answer reads well. It's far weaker at knowing whether the agent took the right action in a payroll run or a billing workflow, and it drifts when the product underneath it changes. As the platform scaled, one judge grading answers stopped being enough.

So I built the quality discipline as a loop, not a verdict. Golden datasets carry the correct answer and the expected actions. Offline judging gets checked against human labels on production samples. Multi-turn simulators, a GenAI bot playing the customer, surface the failures that only appear three turns deep. Regression runs make sure last month's quality is still this month's.

Trust the judge only after you test the judge.

LLM judges are cheap, fast, and necessary. Before trusting them at scale, we validated them against a 200-task human-labeled set.

The judges cleared it. On that evidence we cut a planned 2,000-row vendor labeling dataset we no longer needed: the judges were well tuned, and the spend wasn't justified. An evaluation program should tell you where quality is and where your money isn't required.

Here's a judge run against one golden row. One gate is supposed to fail.

I made the slowest step a review, not a rewrite.

Authoring golden datasets by hand was the slowest step in the loop. A designer opened a blank row and wrote everything: the scenario, the ground truth, the expected actions, the gating criteria.

So I built a pre-population pipeline. An LLM pre-fills the evaluation data first; designers review and correct instead of authoring from blank. It turned blank-page authoring into review-and-correct across 3,000+ rows, without adding headcount, and the same datasets fed new-skill development, not just the eval gate.

It shipped as a pull request through the same review process as platform code: +1,590 lines, reviewed and merged. It's generator-agnostic, discovering writable columns at runtime and reading new fields from a prompt contract, so the prompt could evolve through several versions in six weeks with no code changes. And it runs a dozen automated checks that fail the build like CI: a dropped column, a data leak, drift.

Then a reviewer caught the thing that mattered. The pipeline had a path where it could silently overwrite a human's original ground truth with model output, with no audit trail. I rebuilt the write logic so originals are never touched on any path, added a check that enforces it and fails the run if it's ever violated, and shipped the fix in about four hours.

"This is platform code shipped through the same review pipeline as senior engineering work." from my notes · May 2026

The ground truth had to be right.

The evals are only as trustworthy as the ground truth under them. Designers re-audited expert ground truth as blind KEEP or DELETE calls, original labels hidden so nobody could anchor on the expert's answer: 1,200+ rows in the first wave. If the ground truth is wrong, everything downstream of it is confidently wrong.

Shipping started requiring evidence.

By April 2026, quality scores had risen between 8 and 13 points across three skill domains.

The structural change runs deeper than the lift: a skill didn't ramp until its golden-set results cleared the launch bar. Quality stopped being a report someone wrote and became a gate releases pass through.
