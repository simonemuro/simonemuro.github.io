No. 03 · The Quality Program

The answer sounded right.

I read an answer that was flawless prose and wrong about a date. On an agentic platform that date isn't just words, it's an action taken in someone's books, so I stopped trusting fluent and built a bar that tests for what it hides.

RoleBuilt the eval datasets, tooling, and audits
When2025 – 2026
InstrumentsGolden datasets · LLM judges · simulators · blind audits
OutcomeQuality became a release gate

This is the answer that started the program. The account data is mock, but the failure is the real one: asked whether a customer's federal payroll tax deposit went through, the agent answered in confident, fluent prose, and invented the date it was filed on.

On a screen, a made-up date is a typo you scroll past. In someone's books, it's a confident wrong answer about whether the government got paid. That's the failure I had to design against, and a model that grades how an answer reads will never catch it.

So I built it as a loop that runs on every release, not a report someone writes once. Golden datasets carry the correct answer and the expected actions. Offline judging gets checked against human labels on production samples. Multi-turn simulators, a GenAI bot playing the customer, surface the failures that only appear three turns deep. Regression runs make sure last month's quality is still this month's.

Trust the judge only after you test the judge.

LLM judges are cheap and fast, so we leaned on them. Before trusting them at scale, we validated them against a 200-task human-labeled set.

The judges cleared it. On that evidence we cut a planned 2,000-row vendor labeling dataset we no longer needed: the judges were well tuned, and the spend wasn't justified. An evaluation program should tell you where quality is and where your money isn't required.

I made the slowest step a review, not a rewrite.

Authoring golden datasets by hand was the slowest step in the loop. A designer opened a blank row and wrote everything: the scenario, the ground truth, the expected actions, the gating criteria.

So I built a pre-population pipeline. An LLM pre-fills the evaluation data first; designers review and correct instead of authoring from blank. It turned blank-page authoring into review-and-correct across 3,000+ rows, without adding headcount, and the same datasets fed new-skill development, not just the eval gate.

It shipped as a pull request: +1,590 lines, reviewed and merged. I built it so the generator's prompt could change through several versions in six weeks without anyone touching the code, and a dozen automated checks fail the build like CI the moment a column drops, data leaks, or the set drifts.

Then a reviewer caught the thing that mattered.

The pipeline had a path where it could silently overwrite a human's original ground truth with model output, no audit trail. That's the kind of bug that doesn't announce itself; it just quietly poisons the data every downstream judge depends on. So I didn't guard the path, I deleted it: I rebuilt the write logic so originals are never touched on any path, added a check that enforces it and fails the run if it's ever violated, and shipped the fix in about four hours.

I made the audit blind.

The evals are only as trustworthy as the ground truth under them, so I had the experts' own ground truth re-audited blind: original labels hidden, every row a fresh KEEP or DELETE call, so nobody could anchor on the expert's first answer. 1,200+ rows in the first wave. If the ground truth is wrong, everything built on it inherits the error.

Shipping started requiring evidence.

By April 2026, scores were up 8 to 13 points across three of the skills.

The structural change runs deeper than the lift: a skill didn't ramp until its golden-set results cleared the launch bar. Quality stopped being a report someone wrote and became a gate releases pass through.

A year spent defining what good looks like is why I now build to that bar myself.
